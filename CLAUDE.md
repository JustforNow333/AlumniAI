# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

AI Spreadsheet Analyst: upload a CSV/XLSX, preview it, and ask natural-language questions answered by a fixed whitelist of safe pandas operations. Flask backend (`backend/`) serves a no-build React 18 + Babel-standalone frontend (`frontend/`) from the same origin at `http://localhost:5000`; API routes live under `/api/*`.

## Commands

```bash
# Start the app (backend + frontend together)
./start_app.sh            # Windows: start_app.bat

# Backend tests (run from project root; pytest.ini sets testpaths/pythonpath)
python -m pytest -q
python -m pytest backend/tests/test_analysis_toolkit.py -q          # one file
python -m pytest backend/tests/test_api.py::test_health -q          # one test

# Frontend tests (dependency-free Node test runner)
node --test frontend/tests/api-preview.test.mjs

# Evals (run from backend/)
./venv/Scripts/python.exe -m evals.run_evals --mode offline
./venv/Scripts/python.exe -m evals.run_evals --mode offline --category industry_classification
```

The backend virtualenv is a **Windows** venv at `backend/venv/Scripts/python.exe`. From WSL, run tests with `./backend/venv/Scripts/python.exe -m pytest -q`. Dependencies: `pip install -r backend/requirements.txt`.

`OPENAI_API_KEY` in `backend/.env` is optional — without it the app uses deterministic heuristic intent inference and deterministic answer formatting. Tests never need a key (they set `ai_service.client = None`).

## Security Notes

- Flask sets `SECRET_KEY` from `FLASK_SECRET_KEY` when provided, otherwise generates a random runtime key. Do not hard-code a secret.
- Default runtime host is `127.0.0.1`; set `FLASK_HOST` only when intentionally exposing the app beyond localhost.
- All responses receive `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and `X-XSS-Protection: 1; mode=block`.
- `GET /api/debug/classify-row` is development-only and must stay hidden behind `FLASK_DEBUG` so classifier internals are not exposed in normal runs.
- Dataset registry `file_path` values are resolved only inside configured upload/data storage roots. Preserve the containment checks in `dataset_store._resolve_dataset_file_path`.
- Offline security coverage lives in `backend/tests/test_app_security.py`, plus path-containment coverage in `backend/tests/test_dataset_store.py`. These tests do not require OpenAI and do not involve eval/generated-code execution.

## Architecture

### Ask pipeline (the core flow)

`POST /api/ask` in `backend/app/routes/chat_routes.py` runs a strict inference → plan → execute → present pipeline. The LLM never executes code or computes numbers; it only infers intent and formats results.

1. `dataset_store.load_dataset_dataframe` reloads the full DataFrame from disk by `dataset_id`.
2. `analysis_toolkit.build_dataset_context` builds a compact context (column names/types, missing/unique counts, sample values, 5 sample rows — never the full dataset).
3. `analysis_intent.infer_analysis_intent` produces a semantic intent JSON, via OpenAI when configured, else `heuristic_intent`. Intents use *semantic* column names (`occupation`, `employer`, …) resolved to actual DataFrame columns through exact/case-insensitive/normalized/synonym/fuzzy matching. A `CONCEPT_LIBRARY` (`tech_related`, `software_engineer_role`, `tech_company`) expands fuzzy concepts into search terms. Broad people questions never error out: `alumni_tech_fallback_intent` handles "alumni in tech" phrasing, and `industry_taxonomies.classify_people_question` + `people_filter_fallback_intent` handle every other answerable industry/employer/occupation query (both in `heuristic_intent` and when the LLM is unsure or asks for clarification).
4. `analysis_intent.intent_to_analysis_plan` converts the intent into whitelisted operations (max 3 per request, see `ALLOWED_OPERATION_TYPES` in `analysis_toolkit.py`).
5. `analysis_executor.execute_analysis_plan` validates and runs each operation via `analysis_toolkit.execute_operation`. No `eval`/`exec`/generated code is ever accepted. Unknown operations/columns return structured `{"status": "error"}` results.
6. `answer_presenter.present_answer` formats results into a structured answer (`title`, `summary`, `blocks`, `followups`; block types: `markdown`, `table`, `metrics`, `ranked_list`), falling back to `answer_schema.deterministic_answer_from_results` when the model is unavailable or returns invalid JSON. `answer_schema.normalize_answer` strips HTML, caps sizes, and rejects unknown block types.

### Industry taxonomy people filter

People/alumni queries run through a reusable taxonomy + classification system, not tech-specific code. **Core principle: a broad keyword hit only makes a row a candidate; strict query-aware classification decides final inclusion.** Raw keyword hits must never be presented as the answer count.

- `industry_taxonomies.py` defines editable taxonomies for tech, consulting, investment_banking, banking, finance, healthcare, law, education, media, nonprofit, startups, venture_capital, private_equity, marketing, operations, and government_legal. Each taxonomy has `aliases`, `title_keywords`, `generic_title_keywords` (roles that never confirm alone), `employer_keywords`, `known_companies`, `exclusion_keywords`, `ambiguous_keywords`, `confidence_threshold`, and optional `retrieval_keywords` — broad candidate-generation terms that never confirm a match on their own. The tech taxonomy merges `known_tech_companies.json` (the editable known-company config). Companies may belong to several industries (Spotify is tech *and* media).
- `classify_people_question` maps a question to a filter spec with `filter_type: industry | employer | occupation`. Longest alias wins, `INDUSTRY_PRIORITY` breaks ties (investment_banking > banking > finance, media > tech). A capitalized company after "at" or "for" is treated as an exact employer filter, not as an industry guess. "Who are founders?" → occupation filter; "startup founders" → startups taxonomy. Industry specs also carry query modifiers: `industries`, `required_industries` ("finance consulting" → consulting ∩ finance), `excluded_industries` ("finance but not banking"), `include_adjacent`, `include_functions`, and `query_scope` (`industry`, `technical_role`, `subindustry`, `industry_exclusion`).
- `people_classifier.py` (`classification_version: multi_label_v1`) is the query-aware multi-label classifier used for all industry filters. Each candidate gets an independent profile — `employer_industry` (array: an employer can be technology *and* media), `job_function` (array: consulting_advisory, finance_investing, internal_strategy, product, operations, legal, engineering, …), `specialties` (risk_consulting, transaction_advisory, valuation, restructuring, investment_banking, wealth_management, …) — then a per-query verdict: `classification: direct_match | adjacent | uncertain | non_match` plus `count_as_match`, `confidence`, `internal_reason`, and stable metadata (`match_category`, `match_confidence`, `role_signal`, `employer_signal`, `match_reason_code`). Only direct matches count by default; adjacent/uncertain/non_match rows never count unless the query explicitly asks for adjacent rows, and confidence can never promote a non-direct label. Dedicated deterministic policies exist for tech, investment_banking, banking, finance, marketing, operations, government_legal, and consulting. Other industries wrap `match_row_to_industry`; add new industries by registering a policy in `_INDUSTRY_POLICIES`. The LLM is consulted only for rows deterministic rules leave `uncertain`, and it must not override high-confidence deterministic positives into exclusions.
- Finance/banking split: investment_banking is separate from broad banking and requires explicit investment-banking title wording; known bank employer alone is not enough for investment banking. Broad banking counts bank/advisory employers *and* banking titles (investment banking, corporate/commercial banking, wealth management, M&A, capital markets, sales & trading, restructuring), so broad banking is a superset of investment_banking. Broad banking is separate from finance (asset managers, hedge funds, PE). Government/development institutions (e.g. World Bank) are never treated as banking *or* finance even though their name contains "bank". Finance queries exclude direct banking/investment-banking matches when the query or policy requires it, while weak adjacent excluded matches do not remove rows.
- Tech scope split: broad `query_scope: industry` for "work in tech" is high recall: direct match means explicit technical role OR clearly tech employer. Non-engineering roles at known/strong tech employers (product, growth, strategy, finance, legal, sales, customer success, professional services, operations, government solutions, account management) count as direct for broad discovery queries. `query_scope: tech_company` is employer-first and counts any role at clearly tech employers, while technical roles at non-tech employers can be adjacent. `query_scope: technical_role` is strict and requires software/engineering/technical implementation wording; product or business roles at tech companies are not direct for software-engineer-only queries.
- `industry_matching.match_row_to_industry(occupation, employer, taxonomy)` is the layered per-taxonomy engine: title keyword → known company → employer keyword → exclusion context → optional budgeted LLM employer classifier (confirms only at/above `confidence_threshold`; no-op without an API key) → ambiguous → excluded. Generic business roles (founder, head of growth, CEO, …) count only with a matching employer. Returns `{status: confirmed|uncertain|excluded, match_sources, confidence, internal_reason}`.
- `column_resolver.py` maps messy source columns (`LastName`, `LinkedinURL`, …) to canonical person fields; visible headers for people results are always `First Name`, `Last Name`, `Occupation`, `Employer`, `LinkedIn URL` (never `Nickname` when first/last exist).
- Execution: `contains_any` with `filter_mode: "people"` (new) or `"tech_people"` (legacy, routes through the tech taxonomy) runs `_people_filter_result` in `analysis_toolkit.py`: broad keyword retrieval finds candidates, the classifier decides inclusion, people are deduplicated, and the people_filter shape is returned (`intent`, `entity`, `filter_type`, `industry`, `criteria_label`, `answer_label`, `total_dataset_rows`, `total_keyword_hits`, `total_matches`, `displayed_count`, `display_limit`, `scored_result_count`, `visible_columns`, dict rows) plus classification/tracing metadata (`raw_candidate_count`, `candidate_count`, `direct_match_count`, `adjacent_not_counted_count`, `uncertain_not_counted_count`, `excluded_count`, `target_industries`, `excluded_industries`, `taxonomy_used`, `display_columns`, `classification_version`). `operation_results[*].rows` contains the full scored direct-match set for eval/scoring; `direct_rows`, `adjacent_rows`, `uncertain_rows`, and `row_sections` carry the surfaced buckets for high-recall discovery answers. `displayed_count` is the UI cap. `total_matches` is always the headline direct count. Match reasons/confidence/multi-label profiles live under a separate `debug` key and must stay out of visible tables.
- Even when the LLM confidently returns a generic `find_records` keyword plan, `_plan_find_records` re-checks `classify_people_question` and routes people/industry questions through the strict classifier (`filter_mode: "people"`). This re-check runs on the **original user question** — `infer_analysis_intent` stamps `intent["original_question"]` (the verbatim question) so planning never re-classifies the model's rephrased `user_goal`. This matters because the model often rewrites e.g. "Find alumni in banking." into "…banking or investment banking" or "finance but not banking" into "…excluding banking", which would otherwise mis-route the deterministic spec to `investment_banking` or drop the finance target. Offline (`heuristic_intent`) already used the real question, so this keeps hybrid routing identical to offline for finance/banking/exclusion queries.
- Dev-only `GET /api/debug/classify-row?dataset_id=…&name=…&industry=…` explains why a specific person was included/excluded.
- The tech wrappers `is_explicit_technical_title`, `classify_employer_tech_status`, and `is_strong_non_tech_context` in `analysis_toolkit.py` are kept for backward compatibility and delegate to the generic engine.

### Persistence & dataset library

Uploads are stored as `backend/uploads/<uuid>_<filename>`; metadata-only registry in `backend/data/datasets.json` (`dataset_store.py`). DataFrames are re-read from disk per request — there is no in-memory dataset cache. Flask config keys `UPLOAD_FOLDER`, `DATA_FOLDER`, `DATASET_REGISTRY_PATH` let tests point at temp storage.

Dataset library endpoints (`dataset_routes.py`): `GET /api/datasets` lists all saved datasets newest-first (ties broken by registry insertion order) without loading DataFrames — each entry has `dataset_id`, `display_name` (falls back to `original_filename`), counts, `columns`, `file_type`, and `status` (`"ready"` or `"missing"` when the file is gone from disk; never crashes). `PATCH /api/datasets/<id>` renames (`{"display_name": ...}`, non-empty, atomic via the registry tmp-file write). `DELETE /api/datasets/<id>` removes metadata + the stored file (clean JSON 404 when unknown). Registry entries now carry `display_name` (older entries without it are tolerated by `dataset_public_metadata`).

### Saved insights

Manually saved answer snapshots tied to a `dataset_id` (`insight_store.py`, registry at `backend/data/saved_insights.json`, config key `INSIGHTS_REGISTRY_PATH` defaulting under `DATA_FOLDER`). **Insights are saved only when the user clicks Save insight or saves a History item as an insight — `/api/ask` never writes insights.** An insight is a curated snapshot (`insight_id`, `dataset_id`, `dataset_name_snapshot`, `title`, `question`, `answer`, `created_at`/`updated_at`, `tags`, `metadata` with row/column counts, optional `response_payload`): the answer is never recomputed and no DataFrame/file contents are stored. Endpoints (`insight_routes.py`): `GET /api/insights[?dataset_id=…]` newest-first; `POST /api/insights` (validates dataset exists + non-empty question/answer; title auto-generated from the question when missing) → 201; `GET/PATCH/DELETE /api/insights/<id>` (PATCH edits title/tags only — dataset_id/question/answer are immutable; clean JSON 404s). Insights whose dataset was deleted are kept and reported with `dataset_status: "deleted"` (vs `"ready"`) without crashing.

### History

Automatic analysis history snapshots live in `history_store.py`, registry `backend/data/history.json` (config key `HISTORY_REGISTRY_PATH`, defaulting under `DATA_FOLDER`). After `/api/ask` successfully produces a non-empty operation-backed answer, `chat_routes.py` stores a history item with `history_id`, `dataset_id`, dataset filename snapshot, question-derived title, question, `answer_text`, `status: "success"`, timestamps, metadata such as row/column counts and searched columns, and the exact JSON-safe `response_payload` used to reopen the answer. Validation errors, missing datasets, planner failures, operation errors, and empty answers do not create history. History reads tolerate old records with missing/malformed `response_payload`; the stored response renders even if the original dataset is inactive or deleted.

History endpoints (`history_routes.py`): `GET /api/history` newest-first, `GET /api/history/<id>`, `POST /api/history` (manual creation fallback), `DELETE /api/history/<id>`, `DELETE /api/history` clear all, and `POST /api/history/<id>/save-insight` as a backend bridge. The frontend primarily saves a history item as an insight by sending the item snapshot through `POST /api/insights`, avoiding any model re-run.

### Frontend

No build step: `index.html` loads React/Babel from CDN and the `.jsx` files via in-browser transform; everything is exposed on `window` (no modules). `window.ALUMNI_CONFIG = { useApi, apiBase }` toggles backend mode vs. an in-browser CSV demo mode (`engine.jsx` + `sample-data.jsx`, demo only). `api.jsx` is the backend adapter: it normalizes upload/preview/ask responses plus the dataset library (`Alumni.datasets()/renameDataset()/deleteDataset()`, entries normalized by `normalizeDatasetEntry`) and saved insights (`Alumni.insights()/insight()/saveInsight()/renameInsight()/deleteInsight()`, normalized by `normalizeInsightEntry`; helpers `defaultInsightTitle` and `insightTextFromAnswer` flatten a structured answer into the plain-text snapshot), sanitizes structured answers (hides debug columns unless `cfg().debug`, canonicalizes name/LinkedIn columns, uses `total_matches` as the headline stat for people-filter results). `app.jsx` holds the UI and the structured-answer renderers. `frontend/tests/api-preview.test.mjs` tests the adapter by evaluating `api.jsx` in Node — keep `api.jsx` free of JSX/browser-only syntax.

Dataset library UI (`app.jsx`): `App` owns `datasets`, `view` (`"chat" | "datasets" | "insights" | "history"`), the active dataset (`ds`), preview state, saved insights state, and history state. On boot in API mode it fetches `GET /api/datasets`, restores the active dataset from localStorage key `alumniActiveDatasetId` (else newest), and loads its preview; with no datasets it shows the upload empty state. The sidebar `Rail` navigates between Conversations, Datasets, Saved insights, and History; the ACTIVE DATASET item reflects/links to the library. `DatasetLibrary` renders in the main panel: select (activates + loads preview), upload, rename (`window.prompt`), delete (`window.confirm`; switches to the next dataset or clears). `Workspace` is keyed by `dataset_id` + a chat sequence, so switching datasets or "New analysis" starts a fresh conversation; asks always send the active `dataset_id` and are blocked client-side when none is active.

Saved insights UI (`app.jsx`): `AiMsg` shows a Save insight button under completed answers (only with an originating `question`, an answer, an active `dataset_id`, and API mode); it prompts for a title (default generated from the question), shows Saving…/Saved ✓ states, and blocks duplicate saves. `Workspace.send` stamps `question` onto AI messages for this. `InsightsLibrary` renders the list (All insights / Current dataset only filter — current disabled without an active dataset) and `InsightDetail` the full snapshot with an "Open dataset" button that activates the insight's dataset when it still exists; deleted datasets show a "Dataset deleted" badge but never break the view. `App` owns `insights`, `insightsLoading`, `insightsError`, and `selectedInsightId` (App-level so the open detail survives the Workspace remount when switching datasets); the list loads when the view opens and updates in place on save/rename/delete.

History UI (`app.jsx`): `HistoryLibrary` loads when the History view opens via `Alumni.history()`. It shows the empty state "No history yet. Ask a question about a dataset and it will appear here.", cards with question/title, dataset filename/status, timestamp, answer preview, and small metric chips when present, plus Open, Save as insight, Delete, and Clear history. `HistoryDetail` uses the shared `DatasetResponseView` with the stored `response_payload`; opening history never calls `/api/ask`, never activates a dataset, and still renders if the source dataset was deleted. `Alumni.saveHistoryAsInsight(item)` sends the stored snapshot through `Alumni.saveInsight`.

### Result schema conventions

Operation results carry `is_filtered`, top-level counts (`total_rows`, `raw_match_count`, `matched_row_count`, `returned_row_count`, `scored_result_count`, `displayed_count`, `display_limit`), and a `metrics` dict that duplicates them plus backward-compatible aliases (`rows_matched`, `rows_returned`, `searched_columns`). For people filters, `rows` should contain the full scored direct-match set while `displayed_count` tracks the UI cap. High-recall discovery answers may also include `direct_rows`, `adjacent_rows`, `uncertain_rows`, `direct_count`, `adjacent_count`, `uncertain_count`, and `row_sections`; counts must be computed from these structured rows, not parsed from prose. Search operations keep `search_columns` separate from `display_columns` — searched-only columns are not displayed unless the user asked for them. Display limits must never be presented as the answer count.
